## What Vagaro Is

Vagaro is a booking marketplace and business management SaaS for salons, spas, and fitness studios. Founded 2009, reached unicorn status ($1B valuation) in 2021, acquired Schedulicity in 2025. The platform serves both consumers (booking appointments) and businesses (scheduling, POS, CRM, website builder). Four mobile apps: `com.vagaro.vagaro` (consumer), `com.vagaro.vagaropro` (business), `com.vagaro.iconic23` (white-label branded), `com.vagaro.connectbyvagaro` (messaging).

## Architecture

**Frontend/Backend**: ASP.NET Core (Kestrel) on Azure App Service, West US 2. The platform carries a significant ASP.NET WebForms legacy â€” ASMX WebServices (`/WebServices/[service].asmx/[method]`) are still active in production alongside a newer React layer (VagaroConnect). The obfuscated `CallWebServicehed` function present on every page is jQuery AJAX calling these legacy services, using a custom auth header scheme with fields `K` (key), `I` (IV), and `Val`, generated by client-side `getek()`/`getei()`/`EncIV()` functions and stored in hidden inputs `#hdnToken1`, `#hdnToken2`.

**Static assets**: Production CSS and JavaScript bundles load from Azure Blob Storage at `stagebundle.blob.core.windows.net`. Business media (photos, assets per-merchant) comes from Rackspace CDN across 14+ per-business containers in `*.ssl.cf2.rackcdn.com`. Static framework assets use Azure CDN at `vagaro.azureedge.net`.

**Real-time / push**: SignalR for the VagaroConnect messaging layer and Vera Chat. Firebase Cloud Messaging for push notifications (project `semi-final-de6d2`, discussed below).

**Payments**: Stripe (loaded on business detail pages), Affirm (BNPL).

**WAF**: Imperva proxies all primary domains. The TLS certificate Subject CN is `imperva.com` â€” the cert SANs list `api.vagaro.com`, `*.bookitall.com`, `*.zenswipe.com`, `bookitall.com`, `vagaro.com`, and `imperva.com`. Imperva proxy ID 10605-100, error 15 (IP reputation block) is common on residential IPs.

**Domains**: `bookitall.com` is Vagaro's consumer marketplace under an alternative brand (DNS via Azure, registered 2010). `zenswipe.com` (registered 2013, GoDaddy, DNS doesn't resolve) appears dormant â€” possibly an acquired domain tied to their card reader product. `lp.bookitall.com` routes directly to Azure Web App `vagaro-erp-redirect-1.azurewebsites.net` (waws-prod-mwh-007, West US 2, 52.175.254.10), bypassing Imperva entirely.

**Multi-region**: US02 (West US / `wus`), AU1 (Australia). Region is baked into API URLs (`/us02/api/v2/`).

**Support infrastructure**: Zendesk Help Center at `support.vagaro.com` (Cloudflare), UserVoice/Zendesk feedback at `feedback.vagaro.com`, Atlassian Statuspage at `status.vagaro.com`. Status page lists Twilio (SMS), SendGrid v1 and v3 (email), and a standalone credit card processing component.

## Surveillance: Two Different Sites

The surveillance profile differs significantly between the public booking marketplace and the business-facing platform.

**Public marketplace (vagaro.com / bookitall.com)**

Consent mode is configured on every page:

```javascript
gtag('consent', 'default', {
  ad_storage: 'denied',
  analytics_storage: 'denied',
  ad_user_data: 'denied',
  ad_personalization: 'denied',
  wait_for_update: 500
})
```

All advertising and analytics storage denied by default, with a 500ms update window for a consent management platform. No GTM detected on public pages. No Meta/Facebook pixel, no Mixpanel, no Segment. GA4 tag `G-2L62EDJ2V9` is present (same as the Firebase `measurementId`). Pinterest domain verification (`c50a0601b344d295f01870b5e0e87d10`) and Ahrefs site verification are passive â€” no pixel-level tracking.

On first load, an IP geolocation lookup sets the `proximitystate_v3` cookie containing city, state, zip, country, timezone offset, and currency symbol. The cookie is read by page JavaScript to route visitors to the appropriate regional subdomain (`/en-gb/`, `/en-ca/`, `/en-au/`). The cookie is set before any user interaction.

## API Architecture

**Enterprise API V2** is publicly documented at `docs.vagaro.com` (ReadMe.io/SuperHub platform). Authentication requires an API key (`x-api-key` header) plus an access token from `/generate-access-token`. Resources: Employees, Locations, Appointments, Customers.

Webhook payloads include full PII without field-level encryption: name, email, phone, address, birthdate, payment amounts, tip amounts, discount data. `FormResponse` webhooks carry data from intake forms â€” spas and health-focused businesses collect health conditions and allergy information via these forms.

**Legacy ASMX WebServices** are still active in production:
